# Ringo 普攻弹道垂直切片 — 设计文档

日期：2026-07-06

## 1. 一句话目标

把 Ringo 的普攻从"一团静态贴片"改造成一条**真正会动的粒子弹道**：枪口闪光 → 子弹从枪口飞向前方目标 → 命中爆开，视觉上做到和真实游戏画面**肉眼接近**。用它作为整个"技能特效还原"方向的方法验证（垂直切片），跑通一次完整数据管线，再谈规模化。

## 2. 现状（已亲眼核实，非文档转述）

- **模型/材质/动画线是好的**：Ringo 模型、贴图、普攻动作在查看器里显示正常。
- **特效线走错了方法**：app.js 里真正生成特效几何的函数只有一个 `createRuntimeEffectPreviewObject`（约在 9073 行），它把 PFX 的 surface 贴图当成**静态面片**贴在角色身上。屏幕上就是一团模糊的红褐色卡片——不是枪口火光，也不是飞行子弹。
- **没有粒子系统**：全项目 app.js 里 `Points`/`PointsMaterial`/`Sprite`-based 粒子渲染命中为 0。
- **子弹没有"发射链路"**：现有诊断自述"3 个定义 projectile / 8 个索引 projectile PFX（无发射链路）"——知道有子弹定义，但从未让子弹真正飞起来。
- **根因**：Codex 给自己设了"没有 live runtime 抓帧证据就不许接管渲染"的硬门槛，于是把精力全投在静态逆向和诊断报告上（135 张调试截图、几十个 audit 脚本），从未建一个真正的粒子系统。

## 3. 为什么从 Ringo 普攻切入

- Ringo 普攻的三段特效在资源里命名清晰、语义明确，正好是一条完整弹道：
  - `RingoAttackMF.pfx` — **Muzzle Flash 枪口闪光**（发射）
  - `RingoAttackShot.pfx` — **飞行子弹**（飞行）
  - `RingoAttackImpact.pfx` — **命中特效**（命中）
- 一个英雄的一个动作，就能同时验证"弹道飞行"和"粒子特效"两块能力。
- 原料齐备且已在本地：
  - 原始 pfx 二进制：`extracted/build_resources_by_path/Effects/Ringo/attack/*.pfx`
  - 特效贴图已解成 PNG：`extracted/effect_textures_preview/Effects/Ringo/attack/*/*.png`
  - pfx 解析结构（references、surfaceRecords、emitterRuntimeProfile、sampledFloats）已在 `effect-pfx-resource-manifest.json`

## 4. 方法：真实参照的"起步坡道"（先走 L1）

本切片只走 **L1**，把方法跑通；L2/L3 是后续对同一效果的精修，不在本切片范围。

- **L1 起点**：用已有的贴图 PNG + pfx 结构 + 采样浮点值，在 Three.js 里建一个真正的粒子/精灵系统，做出第一版会动的弹道。
- **L1 校准**：pfx 里采样出的浮点参数（如 `RingoAttackShot` 的 offset 217=-90、213=0.1）语义尚未标定——哪个是初速度、生命周期、尺寸、颜色，靠**真实游戏画面对照**来定。真实画面来源：①我上网找 Ringo 普攻的技能演示视频；②必要时请你录一小段。
- **L2/L3（后续，不在本切片）**：Android RenderDoc / iOS Metal 抓帧，拿精确参数把效果校准到"准"乃至"逐字节"。

## 5. 范围

**做（Do）：**
- 只做 Ringo 普攻（attack）的 MF → Shot → Impact 三段。
- 在 Ringo 前方固定距离放一个默认目标点/半透明靶，子弹朝它飞、在它身上命中。目标只是一个坐标，不涉及伤害/AI/命中判定。
- 让效果跟随普攻动作的时序播放（发射时枪口闪光、子弹飞出、到达时命中）。

**不做（YAGNI）：**
- 不做其他英雄、其他技能（大招火球留作切片成功后的第二个目标）。
- 不做 L2/L3 抓帧管线（本切片不依赖它）。
- 不碰战斗逻辑（伤害、命中判定、目标 AI）。
- 不动现有 `createRuntimeEffectPreviewObject` 的行为，避免破坏其他英雄现有显示。
- 不用调亮度/曝光等投机手段凑视觉。

## 6. 架构

- **新建独立粒子模块**（如 `extracted/viewer/effect-particles.js`），与现有 `createRuntimeEffectPreviewObject`（面片逻辑）**并存**，不改造、不替换现有逻辑。理由：粒子系统与面片贴图是两种范式，硬改现有代码易重蹈"半成品接管主渲染导致模型消失/层级错乱"的覆辙（handoff 明确警告过）。
- 模块职责边界清晰：
  - **输入**：一个 pfx 的解析记录（贴图 PNG 路径、采样参数、生命周期/变换/颜色 offsets）+ 发射起点（枪口骨骼/挂点）+ 目标落点。
  - **输出**：一组随时间演化的 Three.js 粒子/精灵对象，挂到场景里。
- 先只被 Ringo 普攻这一条路径调用，验证通过后再考虑推广。

## 7. 成功标准（可验证，以视觉为准）

在查看器里选中 Ringo、播放普攻，应当能看到：
1. 发射瞬间枪口有一团**闪光**（而不是静态卡片）。
2. 一颗**子弹从枪口飞向前方靶**，带该有的拖尾/形态。
3. 子弹到达时**爆出命中特效**。
4. 这三段的**颜色、形态、时序**与真实游戏画面（找到的演示视频/你录的片段）**肉眼接近**。

验证方式：截图与真实画面并排对照。**不写只能验证当前近似的结构测试**；测试只用于防止脚本/加载崩坏的下限。

## 8. 风险与待验证

- **pfx 采样参数语义未完全标定**：靠 L1 画面校准补齐；这是切片要暴露和解决的核心难点。
- **L2/L3 抓帧能否挂上该引擎**：有反调试风险，需实测；但 L1 不依赖它，不阻塞本切片。
- **社区服务器依赖**：抓真实画面/后续抓帧依赖社区 CE 服务器在线，不完全可控——这是外部风险，非本切片阻塞项（L1 用现成视频即可起步）。

## 9. 需要你配合的

- **起步阶段：不需要你做任何事**，我自己找演示视频 + 读本地数据先做出一版。
- **校准阶段：可能需要你在能玩的设备上录一小段 Ringo 普攻**（最好慢一点/正对镜头），只录这一个动作即可，用来核对我做的对不对。
